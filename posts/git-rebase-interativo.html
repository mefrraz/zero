Git Rebase Interativo: A Arte de Reescrever a História

Commits desarrumados não têm de ficar assim. O rebase interativo é a tua ferramenta de edição da história.

O Git é uma máquina do tempo. E como qualquer viajante temporal responsável, deves saber que a história pode ser reescrita — desde que ninguém a tenha visto ainda.

O git rebase -i (interativo) é a tua ferramenta de edição. Permite reorganizar, combinar, editar, ou eliminar commits antes de os partilhares.

Porquê Reescrever?

Imagina este histórico durante o desenvolvimento:

3a5f2c1 fix typo
b2e8d4f fix another typo
1c7a3e2 WIP: trying something
8d4b2a9 Add user authentication
f1e6c8d forgot to add file
5a2b7c3 Initial commit

Isto é normal quando estás a trabalhar. Mas antes de fazer push para a branch principal ou abrir um PR, podes (e deves) limpar:

a1b2c3d Add user authentication
5a2b7c3 Initial commit

Limpo. Legível. Conta uma história clara.

O Básico: Como Iniciar

# Rebase interativo dos últimos 5 commits
git rebase -i HEAD~5

# Ou desde um commit específico
git rebase -i abc123

# Ou desde a base de uma branch
git rebase -i main

O Git abre o teu editor com algo assim:

pick 5a2b7c3 Initial commit
pick 8d4b2a9 Add user authentication
pick f1e6c8d forgot to add file
pick 1c7a3e2 WIP: trying something
pick b2e8d4f fix another typo
pick 3a5f2c1 fix typo

Cada linha é um commit. A palavra pick à esquerda é o comando. É aqui que a magia acontece.

Os Comandos

Comando Abreviação O Que Faz
pick p Mantém o commit como está
reword r Mantém o commit, mas edita a mensagem
edit e Para no commit para edições
squash s Combina com o commit anterior, junta mensagens
fixup f Combina com o anterior, descarta a mensagem
drop d Remove o commit completamente

Squash: Combinar Commits

O squash junta um commit ao anterior, combinando as mensagens:

pick 8d4b2a9 Add user authentication
squash f1e6c8d forgot to add file
squash 1c7a3e2 WIP: trying something

Resultado: um único commit com uma mensagem que tu decides.

Fixup: Squash Silencioso

O fixup é como o squash, mas descarta a mensagem do commit combinado:

pick 8d4b2a9 Add user authentication
fixup f1e6c8d forgot to add file
fixup b2e8d4f fix another typo
fixup 3a5f2c1 fix typo

Resultado: um único commit "Add user authentication", limpo.

>
Dica: Podes criar commits de fixup automaticamente: git commit --fixup abc123. Depois, git rebase -i --autosquash organiza-os automaticamente.

Reword: Apenas Editar a Mensagem

reword 8d4b2a9 Add user authentication

O Git para e deixa-te editar a mensagem do commit.

Edit: Parar Para Modificações

edit 8d4b2a9 Add user authentication

O Git para neste commit. Podes: - Modificar ficheiros - Fazer git add - git commit --amend para alterar o commit - git rebase --continue para prosseguir

Reordenar Commits

Simplesmente muda a ordem das linhas:

# Antes
pick a1 Commit A
pick b2 Commit B
pick c3 Commit C

# Depois (C vem primeiro)
pick c3 Commit C
pick a1 Commit A
pick b2 Commit B

Workflow Real

Eis como uso o rebase interativo no dia-a-dia:

1. Desenvolvimento Livre

Durante o desenvolvimento, faço commits frequentes e desarrumados:

git commit -m "WIP"
git commit -m "fix"
git commit -m "actually fix"
git commit -m "add tests"

2. Antes do Push

Limpo a história:

git rebase -i main

Combino tudo em commits lógicos:

pick abc123 Add user authentication
fixup def456 WIP
fixup ghi789 fix
fixup jkl012 actually fix
squash mno345 add tests

3. Mensagens Claras

Aproveito o reword para escrever mensagens descritivas:

feat(auth): add user authentication

- Implement JWT token generation
- Add login/logout endpoints
- Include password hashing with bcrypt
- Add unit tests for auth service

Quando NÃO Fazer Rebase

>
Aviso: Nunca faças rebase em commits que já partilhaste com outros. Se alguém já fez pull do teu trabalho, reescrever a história causa conflitos graves.

Regra de ouro: Rebase na tua branch local antes do push. Depois do push, merge.

# Seguro: rebase antes do primeiro push
git rebase -i main
git push origin feature-branch

# Perigoso: rebase depois de já ter feito push
git rebase -i main
git push --force  # Isto pode destruir trabalho de outros

Se precisares mesmo de forçar após rebase:

# Usa force-with-lease: falha se a remote mudou
git push --force-with-lease

Resolver Conflitos Durante Rebase

Se houver conflitos:

# 1. Resolve os conflitos nos ficheiros
# 2. Marca como resolvido
git add ficheiro-resolvido.py

# 3. Continua o rebase
git rebase --continue

# Ou aborta se algo correr mal
git rebase --abort

Configurações Úteis

# Editor para rebase (vim, code, nano...)
git config --global core.editor "vim"

# Rebase automático em pull
git config --global pull.rebase true

# Autosquash por defeito
git config --global rebase.autoSquash true

Conclusão

O git rebase -i é uma das ferramentas mais poderosas do Git. Permite-te:

  • Manter históricos limpos e legíveis
  • Contar uma história clara do desenvolvimento
  • Corrigir erros antes de os partilhares

Trabalha desarrumado. Commita frequentemente. Mas antes de partilhares, limpa a história.

O Git é uma máquina do tempo. Usa-a responsavelmente.
