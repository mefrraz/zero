Git Rebase vs Merge: A Guerra Que Ninguém Pediu

Merge cria histórico honesto. Rebase cria histórico bonito. Qual escolher?

A internet está dividida. Uns juram por git merge, outros vivem de git rebase. A verdade? Ambos têm razão (e ambos estão errados).

Git Merge

O mais direto: junta duas branches mantendo o histórico completo.

git checkout main
git merge feature-branch

Prós: - ✅ Preserva contexto histórico - ✅ Mostra quando features foram integradas - ✅ Mais seguro (não reescreve histórico)

Contras: - ❌ Histórico "poluído" com merge commits - ❌ Gráfico de commits fica confuso

Git Rebase

Reescreve o histórico, movendo commits para o topo:

git checkout feature-branch
git rebase main

Prós: - ✅ Histórico linear e limpo - ✅ Fácil de entender no log - ✅ Bisect funciona melhor

Contras: - ❌ Reescreve histórico (perigoso em branches públicas) - ❌ Conflitos podem acontecer múltiplas vezes

A Regra de Ouro

Nunca faças rebase em branches públicas!

Se mais alguém está a usar a branch, usa merge. Rebase é para o teu trabalho local.

O Meu Workflow

# Feature branch local? Rebase
git pull --rebase origin main

# Merge para main? Usa merge
git merge --no-ff feature-branch

Conclusão

Não é sobre qual é "melhor". É sobre contexto:

  • Rebase: Limpeza antes de mergear
  • Merge: Integração de features

Usa ambos. A tua equipa agradece.
