Docker Multi-Stage Builds: Porque o Peso Importa

Imagens de 1GB para uma API simples? Multi-stage builds são a solução que devias estar a usar.

Já viste imagens Docker de 1GB para aplicações que ocupam 50MB? Eu também. E é um problema maior do que parece.

Imagens pesadas significam deploys lentos, mais custos de storage, e superfícies de ataque maiores. A solução existe há anos, mas continua subutilizada: multi-stage builds.

O Problema: Uma Imagem, Tudo Incluído

Considera este Dockerfile típico para uma aplicação Python:

FROM python:3.12

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .

CMD ["python", "app.py"]

Parece inofensivo. Mas vamos ver o tamanho:

docker build -t minha-app .
docker images minha-app
# REPOSITORY   TAG       SIZE
# minha-app    latest    1.2GB

1.2GB para uma aplicação que, provavelmente, tem menos de 10MB de código.

O problema é que a imagem python:3.12 inclui tudo: compiladores, headers, ferramentas de build — coisas que precisas para instalar dependências, mas não para executar a aplicação.

A Solução: Multi-Stage Builds

Multi-stage builds permitem-te usar múltiplos FROM no mesmo Dockerfile. Constróis num ambiente completo, mas copias apenas o necessário para a imagem final.

# Stage 1: Build
FROM python:3.12 AS builder

WORKDIR /app
COPY requirements.txt .
RUN pip install --user -r requirements.txt

# Stage 2: Runtime
FROM python:3.12-slim

WORKDIR /app

# Copiar apenas as dependências instaladas
COPY --from=builder /root/.local /root/.local
COPY . .

ENV PATH=/root/.local/bin:$PATH

CMD ["python", "app.py"]

Resultado:

docker images minha-app
# REPOSITORY   TAG       SIZE
# minha-app    latest    187MB

De 1.2GB para 187MB. Uma redução de 85%.

Anatomia de um Multi-Stage Build

Vamos dissecar o que acontece:

# Primeiro stage: nome arbitrário 'builder'
FROM python:3.12 AS builder

O AS builder dá um nome ao stage. Podes ter quantos stages quiseres.

# Segundo stage: imagem final
FROM python:3.12-slim

Este é o stage final. Só o que está aqui vai para a imagem resultante.

# A magia: copiar de outro stage
COPY --from=builder /root/.local /root/.local

O --from=builder permite copiar ficheiros de stages anteriores. Tudo o resto do builder é descartado.

>
Nota: Podes também copiar de imagens externas: COPY --from=nginx:alpine /etc/nginx /etc/nginx

Exemplo Prático: Node.js

O JavaScript tem o mesmo problema. node_modules para build vs runtime:

# Stage 1: Build
FROM node:20-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

# Stage 2: Runtime
FROM node:20-alpine

WORKDIR /app

# Só precisamos dos artefactos de build
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules

ENV NODE_ENV=production
CMD ["node", "dist/index.js"]

Para uma aplicação React ou Next.js, podes ir ainda mais longe com um servidor estático:

# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Servir ficheiros estáticos
FROM nginx:alpine
COPY --from=builder /app/build /usr/share/nginx/html

Resultado típico: de 1GB+ para menos de 50MB.

Comparação de Tamanhos

Aplicação Sem Multi-Stage Com Multi-Stage Redução
Python API 1.2GB 187MB 85%
Node.js API 1.1GB 210MB 81%
React SPA 1.4GB 43MB 97%
Go Binary 850MB 12MB 99%

O Go é particularmente impressionante porque compila para binários estáticos:

# Build
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o main .

# Runtime: scratch é uma imagem vazia
FROM scratch
COPY --from=builder /app/main /main
CMD ["/main"]

12MB. A imagem contém literalmente apenas o binário.

Padrões Avançados

Stages Paralelos Para Testes

FROM node:20-alpine AS base
WORKDIR /app
COPY package*.json ./
RUN npm ci

FROM base AS test
COPY . .
RUN npm run test

FROM base AS build
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html

O stage de testes não afecta a imagem final, mas falha o build se os testes falharem.

Cache Inteligente

Ordena os COPY do menos para o mais frequentemente alterado:

# Muda raramente
COPY package*.json ./
RUN npm ci

# Muda frequentemente
COPY . .
RUN npm run build

Isto maximiza o cache de layers do Docker.

Quando Usar

Usa multi-stage quando: - A aplicação precisa de ferramentas de build que não são necessárias em runtime - Queres minimizar a superfície de ataque (menos binários = menos vulnerabilidades) - Deploy speed importa (imagens menores = pulls mais rápidos) - Custos de storage são relevantes

Podes não precisar quando: - Estás em desenvolvimento local e rebuild speed é prioritário - A imagem base já é mínima - Precisas de debugging tools na imagem final (temporariamente)

Conclusão

Multi-stage builds são uma das funcionalidades mais poderosas e subutilizadas do Docker. Com algumas linhas extra, reduzes drasticamente o tamanho das tuas imagens.

Menos bytes. Deploys mais rápidos. Menos vectores de ataque.

O peso importa. Torna as tuas imagens mais leves.
